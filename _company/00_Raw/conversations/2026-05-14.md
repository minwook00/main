# 💰 수익화 모델별 가격 민감도 분석 프레임워크 (Price Sensitivity Framework)

이 프레임워크는 고객의 **'고민의 심각성(Pain Point Severity)'**과 **'해결에 대한 기대치(Expectation of Solution)'**를 측정하여, 각 수익화 모델에 최적화된 가격 포지셔닝을 결정하는 데 사용됩니다.

## 1. 핵심 입력 변수 (Input Variables from S-A-A Data)

Researcher는 수집한 고객 사례 데이터(S-A-A 구조화 결과)에서 다음 세 가지 차원의 데이터를 추출해야 합니다.

| 변수명 | 정의 | 측정 목표 |
| :--- | :--- | :--- |
| **P_Severity (고민 심각도)** | 고객이 현재 겪는 문제의 감정적/실질적 고통 수준 (1~10점) | 가격에 대한 '필요성'을 결정 |
| **V_Value (가치 인식)** | 고객이 해당 해결책(상담/정보)에서 얻고자 하는 궁극적인 가치 (객관적 가치 vs. 주관적 가치) | 가격의 '정당성'을 결정 |
| **A_Action (행동 의향)** | 고객이 제시된 행동(구매, 구독 등)에 대해 실제로 취할 의향 또는 장벽 (Likelihood to Act) | 가격 탄력성 측정 |

## 2. 수익화 모델별 분석 프레임워크

각 모델에 따라 입력 변수를 조합하여 민감도를 분석합니다.

### Model 1: 단일 상품 (Single Product - 일회성 구매)

**목표:** 높은 초기 전환율 확보 및 즉각적인 가치 전달
**분석 지표:** **P_Severity**와 **V_Value**의 상관관계 측정

| 분석 항목 | 계산/분석 방법 | 가격 민감도 해석 | 권고 방향 (A/B) |
| :--- | :--- | :--- | :--- |
| **최소 수용 가격 (Floor Price)** | P_Severity(10점) $\times$ V_Value(5점) $\times$ 기본 서비스 비용 | 고객이 감당할 수 있는 최저선. 이보다 낮으면 가치 부족으로 인식. | **A:** P_Severity에 비례하여 가격을 높여 '고통 해결'의 즉각성을 강조한다. |
| **최적 지점 (Optimal Point)** | $\text{P\_Severity} \times \text{V\_Value} + (\text{경쟁사 평균} \times 0.1)$ | 시장 경쟁 상황을 반영하여 가격을 조정하는 민감도 범위. | **B:** P_Severity가 높을수록, V_Value 대비 가격 탄력성(Elasticity)이 낮아져 가격 민감도가 감소한다 (고통 때문에 지불). |
| **가격 민감도** | $\text{P\_Severity} / (\text{V\_Value} \times \text{A\_Action})$ | 고객의 행동 의향 대비 가격 변화에 대한 반응 정도. | **결론:** P_Severity가 높을수록 가격 탄력성이 낮아지므로, 프리미엄 포지셔닝이 용이하다. |

### Model 2: 번들 옵션 (Bundle Options - Tiered Pricing)

**목표:** 다양한 니즈 충족 및 고객의 선택권 제공
**분석 지표:** **A_Action**과 **V_Value**의 조합에 따른 확장성 측정

| 분석 항목 | 계산/분석 방법 | 가격 민감도 해석 | 권고 방향 (A/B) |
| :--- | :--- | :--- | :--- |
| **번들 가치 증분 (Incremental Value)** | $\text{V\_Value}_{\text{Bundle}} - \text{V\_Value}_{\text{Single}}$ | 추가 옵션을 구매할 때 고객이 느끼는 '추가적인 고통 해소'의 크기. | **A:** 번들 구성 시, 단일 상품 대비 가치 증분($\Delta V$)이 최소 30% 이상이어야 한다. |
| **옵션 선택 민감도** | $\text{A\_Action}_{\text{Premium}} / \text{A\_Action}_{\text{Basic}}$ | 프리미엄 옵션을 선택할 때의 행동 의향 변화율. | **B:** 고객이 '최소한의 해결'을 원하면 Basic에 집중하고, '완벽한 해소'를 원하면 Premium으로 민감하게 가격을 올린다. |
| **번들 효율성** | $\text{Bundle}_{\text{Revenue}} / \text{Cost}_{\text{Delivery}}$ | 번들 구성이 고객 만족도 대비 수익성을 얼마나 높이는지 측정. | **결론:** 초기에는 V\_Value 기반으로 옵션을 설계하고, A\_Action에 따라 프리미엄 가격을 책정한다. |

### Model 3: 구독 모델 (Subscription Model - Recurring Revenue)

**목표:** 지속적인 관계 구축 및 장기적 가치 제공
**분석 지표:** **P_Severity**의 지속성(Retention)과 **A_Action**의 유지율 측정

| 분석 항목 | 계산/분석 방법 | 가격 민감도 해석 | 권고 방향 (A/B) |
| :--- | :--- | :--- | :--- |
| **이탈 위험 지수 (Churn Risk Index)** | $\text{P\_Severity}_{\text{Monthly}} / \text{Value}_{\text{Last Month}}$ | 고객이 이탈할 확률을 예측하여 가격 조정의 필요성을 판단. | **A:** 초기 3개월은 낮은 가격으로 진입시켜 관계를 구축하고, 이후 P\_Severity가 해결될 때 구독료 인상을 적용한다. |
| **지속적 가치 (LTV Potential)** | $\text{Retention Rate} \times (\text{Monthly Price})$ | 장기 고객 생애 가치(LTV) 잠재력 측정. | **B:** LTV를 극대화하기 위해, 초기 가격 민감도를 낮추더라도 높은 유지율을 보장하는 '고가 프리미엄 구독' 모델에 집중한다. |
| **구독 전환 비용** | $\text{P\_Severity} \times (\text{Cancellation Friction})$ | 해지 시 발생하는 고객의 심리적 저항(Friction) 측정. | **결론:** 높은 P\_Severity를 가진 고객에게는 '해지 불가능성'을 높이는 방식으로 가격 민감도를 무력화시킨다. |

### 2단계: 매칭 알고리즘 로직 프롬프트 (Node.js/TypeScript)

설계된 스키마를 기반으로, **유클리드 거리**를 활용한 가치관 일치도 계산 및 필터링 로직을 TypeScript로 구현합니다. 성능 최적화를 위한 인덱싱 전략도 포함하겠습니다.

<create_file path="sessions/matching_algorithm.ts">
// 🛠️ Node.js/TypeScript 매칭 알고리즘 파일
import { User, MatchingQueue } from '@prisma/client'; // Prisma 클라이언트가 있다고 가정
import { performance } from 'perf_hooks';

/**
 * 유저 데이터 구조 정의 (DB에서 가져온 객체를 기반으로)
 */
interface UserData {
  id: string;
  economicValue: number;
  parentingValue: number;
  religionValue: number;
  hobbyValue: number;
  contactFreqValue: number;
  communityScore: number;
  location: string;
  smokingStatus: string;
}

/**
 * 유클리드 거리를 계산하는 헬퍼 함수
 * @param u1, u2 두 사용자의 가치관 벡터 (5차원)
 * @returns 두 벡터 간의 유클리드 거리
 */
function calculateEuclideanDistance(u1: UserData, u2: UserData): number {
  const dimensions = [
    u1.economicValue,
    u1.parentingValue,
    u1.religionValue,
    u1.hobbyValue,
    u1.contactFreqValue,
  ];

  const diffs = dimensions.map((val, i) => val - (u2 as any)[i]); // 타입 안전성을 위해 임시 캐스팅 사용
  
  let sumOfSquares = 0;
  for (const diff of diffs) {
    sumOfSquares += diff * diff;
  }
  return Math.sqrt(sumOfSquares);
}

/**
 * 핵심 매칭 알고리즘 함수
 * @param targetUserId 비교 대상 유저 ID
 * @param allUsers 모든 사용자 데이터 리스트
 * @returns 추천된 사용자 배열
 */
export function findMatches(targetUserId: string, allUsers: UserData[]): { matchedUsers: { user: User, score: number }[], performanceInfo: { durationMs: number, index: number } } {
  const targetUser = allUsers.find(u => u.id === targetUserId);

  if (!targetUser) {
    throw new Error(`Error: Target User ID ${targetUserId} not found.`);
  }

  const results: { user: User, score: number }[] = [];
  const startTime = performance.now();

  // 1차 필터링: Hard Constraints (반경 및 흡연 여부)
  const hardConstraintMatches = allUsers.filter(otherUser => {
    // [추측] 실제 GPS 기반 반경 계산은 DB 쿼리나 별도 서비스에서 처리해야 하지만, 여기서는 DB에 저장된 location 문자열을 가정하고 간단히 필터링합니다.
    // 실제 구현 시: Haversine 공식으로 거리 계산 필요.
    const isWithinRange = targetUser.location === otherUser.location || (targetUser.location.startsWith(otherUser.location.substring(0, 5))); // 매우 단순화된 예시 필터링

    const isSmokerMatch = targetUser.smokingStatus === 'No' && otherUser.smokingStatus === 'No'; // 둘 다 비흡연인 경우만 매칭 시도 (예시)

    return isWithinRange && isSmokerMatch;
  });


  // 2차: 가치관 기반 유사도 계산 (Euclidean Distance)
  for (const otherUser of hardConstraintMatches) {
    const distance = calculateEuclideanDistance(targetUser, otherUser);
    results.push({ user: otherUser, score: distance });
  }

  // 3차: 커뮤니티 활동 지수로 최종 순위 결정
  results.sort((a, b) => b.communityScore - a.communityScore);

  // 상위 10명 추출 (실제로는 Hard Constraint 통과 후의 전체 유저 중 가장 유사한 순으로 필터링해야 함)
  const finalRecommendations = results.slice(0, 10);

  const durationMs = performance.now() - startTime;

  return { matchedUsers: finalRecommendations, performanceInfo: { durationMs, index: results.length } };
}

// --- 인덱싱 전략 설명 ---
/*
성능 최적화를 위한 인덱싱 전략:
1. Location 필드에 공간 인덱스 (PostGIS 등) 적용: 50km 반경 검색의 O(N) 탐색을 O(log N)으로 개선합니다.
2. smokingStatus 및 location 필드에 B-Tree 인덱스 적용: 1차 Hard Constraint 필터링 속도를 극대화합니다.
3. communityScore 필드에 B-Tree 인덱스 적용: 최종 순위 결정(3차 필터링) 시 정렬(`ORDER BY`) 연산 비용을 줄입니다.
*/

### 2단계: 매칭 알고리즘 및 인덱싱 전략 (Node.js/TypeScript)

데이터베이스 스키마(`sessions/schema_design.prisma`)를 기반으로, 유클리드 거리 계산을 포함한 핵심 매칭 로직과 성능 최적화 방안을 제시합니다.

#### 1. TypeScript 알고리즘 함수 설계

핵심은 `calculateSimilarity` 함수와 이를 활용하는 메인 추천 로직입니다. 가치관 점수 5가지 항목(`economicValue`, `parentingValue`, `religionValue`, `hobbyValue`, `contactFreqValue`)을 기준으로 유클리드 거리를 계산합니다.

<create_file path="sessions/matching_algorithm.ts">
/**
 * sessions/matching_algorithm.ts
 */

import { User } from './schema_design'; // 임시 타입 정의 (실제 Prisma 모델 참조 필요)

// 가치관 항목의 인덱스 정의 (DB 스키마 기반)
const VALUE_FIELDS = [
    'economicValue',
    'parentingValue',
    'religionValue',
    'hobbyValue',
    'contactFreqValue'
];

/**
 * 유저 두 명 간의 가치관 유사도를 유클리드 거리로 계산합니다.
 * 점수가 높을수록(10점에 가까울수록) 더 잘 맞다고 판단합니다. 거리가 작을수록 유사도가 높습니다.
 * @param userA 첫 번째 사용자 객체
 * @param userB 두 번째 사용자 객체
 * @returns 유클리드 거리 (숫자)
 */
export function calculateEuclideanDistance(userA: User, userB: User): number {
    let sumOfSquaredDifferences = 0;

    for (const field of VALUE_FIELDS) {
        const diff = userA[field] - userB[field];
        sumOfSquaredDifferences += diff * diff;
    }

    // 유클리드 거리: 제곱합의 제곱근
    return Math.sqrt(sumOfSquaredDifferences);
}

/**
 * 매칭 알고리즘 실행 함수
 * @param targetUserId 추천을 받을 기준 유저 ID
 * @param allUsers 전체 사용자 목록 (DB에서 조회된 배열)
 * @returns 최종 추천 리스트
 */
export function recommendMatches(targetUserId: string, allUsers: User[]): { user: User; score: number }[] {
    const targetUser = allUsers.find(u => u.id === targetUserId);
    if (!targetUser) {
        throw new Error("대상 유저 ID를 찾을 수 없습니다.");
    }

    const results: { user: User; score: number }[] = [];

    for (const candidate of allUsers) {
        // 1차 필터링: Hard Constraints 검사
        if (!passesHardConstraints(targetUser, candidate)) {
            continue; // 조건 불만족 시 스킵
        }

        // 2차 계산: 가치관 유사도 계산 (유클리드 거리)
        const distance = calculateEuclideanDistance(targetUser, candidate);

        results.push({ user: candidate, score: distance });
    }

    // 3차 필터링 및 정렬: 커뮤니티 활동 지수 기반 추천
    // 커뮤니티 활동 지수가 높은 순으로 정렬 (점수가 작을수록 유사도가 높으므로)
    results.sort((a, b) => {
        if (a.user.communityScore !== b.user.communityScore) {
            return b.user.communityScore - a.user.communityScore; // 커뮤니티 점수 내림차순 정렬
        }
        return a.score - b.score; // 커뮤니티 점수가 같으면 거리(유사도)가 작은 순으로 정렬
    });

    // 최종 3명 추천
    return results.slice(0, 3);
}

/**
 * Hard Constraints 필터링 함수
 */
function passesHardConstraints(target: User, candidate: User): boolean {
    // 지역 필터 (반경 50km)
    if (Math.abs(target.location.localeCompare(candidate.location)) > 50) { // 실제 거리 계산은 복잡하므로 단순 문자열 비교로 대체하며, 실제 구현 시 지리 정보 라이브러리 필요
        return false;
    }

    // 흡연 여부 필터 (Hard Constraint)
    if (target.smokingStatus === 'No' && candidate.smokingStatus === 'Yes') {
        return false; // 비흡연자만 찾도록 가정하여 필터링 예시
    }
    
    return true;
}

#### 2. 인덱싱 전략 설명 (성능 최적화)

대량의 유저 처리 시, DB에서 필터링과 유사도 계산을 효율적으로 수행하기 위해 다음과 같은 인덱싱 전략을 권장합니다.

1.  **B-Tree Index on Location:** `location` 필드에 B-Tree 인덱스를 설정하여 50km 반경 검색(Geospatial Query)의 속도를 극적으로 향상시킵니다. (PostgreSQL의 PostGIS 확장 사용 권장).
2.  **Composite Index for Filtering:** `smokingStatus`와 `location`을 포함하는 복합 인덱스(`(smokingStatus, location)`)를 생성하여 Hard Constraints 필터링 시 DB 레벨에서 빠르게 후보군을 줄입니다.
3.  **Vector Indexing (선택 사항):** 만약 추후 임베딩 벡터 기반의 유사도 검색(KNN)으로 확장한다면, `value_scores`와 같은 수치형 데이터에 대해 HNSW나 IVFFlat 인덱스를 고려하여 고차원 공간에서의 근접 이웃 검색을 최적화해야 합니다.

### 1. 브랜드 핵심 정의 (Brand Core)

*   **브랜드 이름:** 심리상담 크리에이터
*   **핵심 메시지:** 복잡한 연애/가치관 문제를 명쾌하고 따뜻하게 해결해주는 전문가.
*   **톤앤매너:** 전문적이지만 딱딱하지 않고, 친구처럼 편안하며 공감적임 (Professional yet Friendly & Empathetic).
*   **시각적 목표:** 복잡한 심리적 고민을 '명료한 길'로 안내하는 **신뢰성 있는 나침반** 같은 느낌.

### 2. 컬러 시스템 (Color System)

전문성과 친근함을 동시에 충족시키기 위해, 신뢰감을 주는 딥(Deep) 톤과 따뜻하고 접근하기 쉬운 포인트 색상을 조합합니다.

| **Primary (신뢰)** | Deep Navy (딥 네이비) | `#1A2338` | 전문성, 깊이, 신뢰감. 메인 타이포그래피 및 헤더에 사용. | 안정감, 지성, 전문성 |

### 3. 타이포그래피 시스템 (Typography System)

가독성을 최우선으로 하며, 전문적이면서도 인간적인 느낌을 주는 산세리프 계열을 메인으로 사용합니다.

*   **헤드라인 (H1, H2):** **Pretendard Bold / Black**
    *   용도: 핵심 메시지 전달. 크고 명확하게 사용하여 주목도를 높입니다.
*   **본문 (Body Text):** **Pretendard Medium / Regular**
    *   용도: 긴 심리적 내용을 읽는 부분. 충분한 행간(Line Height)을 확보하여 친절함을 더합니다. (최소 1.6배 이상 권장)
*   **보조 텍스트/강조:** **Spoqa Han Sans Neo / Regular**
    *   용도: 데이터, 인용구 등 보조 정보에 사용하여 세련됨과 깔끔함을 유지합니다.

### 4. 레이아웃 및 구조 제안 (Layout & Structure Proposal)

랜딩 페이지는 사용자가 '고민'에서 '해결책 찾기'로 자연스럽게 이동하도록 설계합니다.

**A. 히어로 섹션 (Hero Section: 신뢰 구축)**
*   **스타일:** 배경은 Light Cream을 사용하여 부드럽게 시작. 중앙에 브랜드의 핵심 가치(슬로건)를 배치하고, 따뜻한 Peach 색상으로 CTA 버튼을 강조합니다.
*   **구성:**
    1.  **H1:** 타깃 고객의 가장 큰 고통(Pain Point)을 직접적으로 언급하며 공감 유도 (예: "당신의 연애 가치관, 혹시 준비가 덜 되었나요?")
    2.  **Subtext:** 브랜드의 전문성과 친근함을 동시에 전달하는 문구.
    3.  **CTA:** 가장 중요한 행동(무료 상담 신청)을 Warm Peach 또는 Soft Teal 색상으로 강조하여 클릭 유도.

**B. 문제 제기 및 공감 섹션 (Pain Point & Empathy: 관계 형성)**
*   **스타일:** 카드형 레이아웃을 사용하여 정보를 분절하고 소화하기 쉽게 만듭니다.
*   **구성:** 사용자들이 흔히 겪는 연애/가치관 관련 고민들을 **'상황(Situation)'**별로 카테고리화하여 제시합니다. 각 카드에는 짧은 공감 문구와 함께, 해당 고민이 왜 중요한지에 대한 심리학적 관점을 간략하게 제공합니다.

**C. 해결책 및 프로세스 섹션 (Solution & Process: 전문성 증명)**
*   **스타일:** 단계별(Step-by-Step) 또는 흐름도(Flowchart) 형태로 디자인하여, 복잡한 상담 과정을 명확하고 예측 가능하게 만듭니다.
*   **구성:**
    1.  **프레임워크 제시:** 'S-A-A (상황-비유-행동)'와 같은 자체 프레임워크를 시각적으로 강조합니다.
    2.  **단계별 설명:** 상담이 어떻게 진행되는지(진단 $\rightarrow$ 비유 $\rightarrow$ 행동 계획 $\rightarrow$ 실천)를 명확하게 보여줍니다. (Soft Teal을 사용하여 프로세스의 흐름을 시각화).

**D. 신뢰 구축 섹션 (Trust Building: 포트폴리오/후기)**
*   **스타일:** 고객의 실제 사례(S-A-A 데이터 기반)나 간단한 후기를 배치하여 전문성을 뒷받침합니다.
*   **구성:** 단순한 텍스트보다, '고객의 변화'를 보여주는 시각적 요소(레이더 차트 또는 성장 그래프 형태의 추상적인 시각화)를 활용하여 **정량적/정성적 성과**를 은유적으로 표현합니다.
